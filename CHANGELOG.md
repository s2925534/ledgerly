- Completed the last two Phase 26 brainstorm items: full-text corpus search. Phase 7 built a SQLite FTS5 index (`fts_index_search`) but nothing anywhere ever queried it — this wires it up. New `ledgerly.engine.database.search_corpus` runs an FTS5 `MATCH`/`bm25()` query and returns ranked results with highlighted snippets; never auto-creates or activates the SQLite index (`status: "not_indexed"` with a `db sync` hint if it doesn't exist yet), matching AGENTS.md's opt-in-cache rule. Found and fixed two real gaps while building this: (1) personal notes/meeting notes/transcripts were entirely missing from the indexed document set despite the TODO item explicitly naming "notes" as one of the four things to cover — added `personal-notes.yaml` to `_fts_documents`; (2) raw FTS5 `MATCH` syntax chokes on ordinary English words containing `-` or `:` (e.g. "self-driving" errors as invalid syntax) — added `_fts_query_terms`, which quotes each word as a literal phrase token so a plain keyword search box never surfaces a raw SQLite syntax error for common punctuation. New `ledgerly search-corpus <query>` CLI command, `GET /api/v1/db/search?query=...` API route (documented in `docs/api/CONTRACT.md`), and a "Search corpus" web panel with a `/` keyboard shortcut to jump to it (added now that the search box it was deferred for actually exists) — fixed a real XSS risk caught before shipping: the SQLite `snippet()` highlight markers were being inserted as raw HTML, which would have let corpus content containing `<script>` execute; fixed by escaping the snippet text first and only then converting the still-intact `[`/`]` markers to `<mark>` tags. 6 new tests (database + API, 412 total), verified live (CLI end-to-end including the hyphenated-word case, a login+render+search API smoke test). - Completed the last unstarted Phase 26 brainstorm item: a short design note (`docs/ZOTERO_SYNC_DESIGN_NOTE.md`) on whether two-way Zotero sync could ever be done safely.
