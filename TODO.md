## Phase 27: AI Output Grounding and Anti-Hallucination Enforcement (foundational — gates Phases 5, 22, 23, 25)

Codifies Pedro's non-negotiable core rule, added to AGENTS.md 2026-07-16: the deterministic core must always work fully with zero AI configured (already true by design — this phase is about the AI-active half), and whenever AI is active, it must never fabricate. Every factual claim an AI feature produces must trace back to the user's real accepted-source text, artefacts, claims, or their own notes/thoughts — never invented. This phase is the actual engineering mechanism behind that rule; every other AI-tagged item in this file (Phases 5, 22, 23, 25) depends on it and should not ship without it.

- [ ] **Deterministic** - Design a grounding-check mechanism: every AI response containing a factual assertion must carry a reference back to the specific source excerpt, claim ID, artefact ID, or note it drew from, so the claim is auditable rather than just asserted.
- [ ] **Deterministic** - Add an explicit "insufficient evidence" response mode across all `engine.ai` functions: when the safe context doesn't support an answer, the function must say so (e.g. `insufficient_evidence: true` in the response envelope, per the `docs/api/CONTRACT.md` Future AI Routes update) rather than generating a plausible-sounding but ungrounded answer. This is a required, successful outcome, not an error path to minimize.
- [ ] **Deterministic** - Provenance tagging for any AI-generated text inserted into a document, artefact, or report: visually/structurally distinguish AI-generated text from user-authored text and from verbatim source quotes, so a human reader can always tell what came from where. Applies to the document-vault edit-session work (Phase 8) and the Phase 25 document-review command alike.
- [ ] **Deterministic** - Build an adversarial regression test suite of "off-corpus" queries (asking about a topic no accepted source in the test workspace covers) for every existing and planned AI feature, asserting each one refuses/flags rather than inventing an answer. Required before any AI feature in Phases 5, 22, 23, or 25 can be considered done, not just at initial build time.
- [ ] **Deterministic** - Add a CI/test check that runs the full deterministic (non-AI) test suite and a smoke pass over `ledgerly doctor`/`ledgerly status`/core CLI commands with every AI-related env var unset, to keep "the tool fully works with zero AI configured" an enforced guarantee rather than an assumption.
- [ ] **Deterministic** - Extend `docs/api/CONTRACT.md`'s Future AI Routes section (already updated 2026-07-16 with an `insufficient_evidence` field placeholder) with the concrete grounding-field shape once one is designed.
- [ ] **Deterministic** - Rework `engine.ai.build_safe_context` (and any future AI context-building function) to retrieve relevant indexed/chunked excerpts instead of truncating each source from the start up to `max_excerpt_chars`. Two building blocks already exist and just need wiring together for this: the SQLite FTS5 index (`fts_index_search`, Phase 7) for keyword-relevant excerpt retrieval given a query/operation, and paragraph/sentence-level derived-text anchors (`engine/derived_text.py`, Phase 8) for precise, individually-citable chunks. Per AGENTS.md's Core Rule item 6 (added 2026-07-16, applies system-wide wherever AI touches long files) — this is both an efficiency win (less irrelevant content sent per request) and a grounding-precision win (a chunk-level anchor is a far more precise citation than a whole-document reference, directly serving Phase 31's per-suggestion source popups).

## Phase 28: Guided Research-Question Elicitation and AI-Assisted Paper Drafting (topic → refined, falsifiable RQ → draft → deterministic review gate)

From Pedro, 2026-07-16: a stated core goal of this project, not just a feature — someone should be able to start from a vague idea ("let's research about X"), be walked through a guided sequence of questions that refines it into one or more proper, falsifiable research questions (defaulting to a claim framed to be proved or disproved, in service of genuinely novel knowledge rather than just describing a topic), and then be able to say "create a research paper to prove or disprove this" and get a draft good enough to genuinely work from — full AI-drafted prose if AI assistance is requested, gated behind a mandatory deterministic review flag either way. Today, research-question capture is a single free-text prompt only available during `init` (`_prompt_research_questions` in `cli.py`) — no refinement, no structure, no way to run it again later, and no path from an approved RQ to an actual paper draft.

- [ ] **Deterministic** - New standalone entry point usable any time, not just during `init`: `ledgerly rq wizard` — captures a free-text topic, then runs a fixed sequence of guiding questions (mixing open-answer and multiple-choice, reusing the existing `_prompt_numbered_choice` pattern already used elsewhere in `cli.py`) covering scope/context, the phenomenon or relationship of interest, question type (descriptive / comparative / causal / evaluative — a standard, deterministic research-methods taxonomy, not invented per-answer), what existing assumption is being tested, and what evidence would count as support vs. refutation.
- [ ] **Deterministic** - Extend the research question data model (`ledgerly.engine.research_questions`) with explicit `hypothesis` (the claim being tested), `question_type`, and `proof_criteria`/`disproof_criteria` fields, populated deterministically from the wizard's answers — this operationalizes "default to a provable-or-disprovable claim" rather than an open-ended topic statement. Purely additive: existing fields (`question`, `subquestions`, `status`) and existing workflows (`rqs list/approve/reject/archive/check/assess`) keep working unchanged on RQs without these fields.
- [ ] **Deterministic** - Wizard output is saved through the existing draft-RQ storage path (same file `research_questions.py` already writes to) — no new storage mechanism, so `rqs check`/`rqs assess`/`rqs approve` work on wizard output exactly as they do on init-time or manually drafted RQs.
- [ ] **Deterministic** - If the user's opening topic answer implies more than one distinct angle, the wizard should be able to propose a small set of candidate research questions rather than forcing one — matching Pedro's "a first research question or set of them" framing — with the user able to keep or discard each candidate before saving.
- [ ] **API/Web** - The same wizard as a multi-step web UI flow (one guiding question at a time), once API routes exist for it — none of today's `/api/v1/rqs/*` routes support an interactive multi-step flow.
- [ ] **Deterministic** - `ledgerly paper draft <rq-id>` (or an extension of the existing `research-question-brief` artefact type in `engine/artefact_creation.py`): a deterministic, AI-free paper *skeleton* assembled from existing artefact types (`research-question-brief`, `literature-review-matrix`, `claim-evidence-table`) into one document-vault target with a standard shape (hypothesis statement, background/lit review, evidence-for/evidence-against sections populated from the real claim ledger, a prove-or-disprove conclusion placeholder) — this is what "must work fully with zero AI configured" means for this feature: a genuinely useful scaffold from real workspace data, not empty prose.
- [ ] **AI** - `ledgerly paper draft <rq-id> --ai`: an AI-assisted tier that fills the deterministic skeleton with actual drafted prose grounded in the workspace's accepted-source excerpts and claim ledger (same `build_safe_context` boundary every other AI route uses), aiming for genuinely submission-quality writing — but every generated paragraph must be traceable per AGENTS.md's "Core Rule: No Hallucinations" (Phase 27), and the command must explicitly flag/refuse sections the corpus doesn't support rather than inventing content to fill a gap.
- [ ] **AI** - Build this on Phase 8's still-open "AI edit sessions" item (the paragraph/sentence anchor infrastructure it needs already exists) rather than a separate drafting mechanism — treat whole-paper drafting as a large-scale instance of the same reviewable-before-applying AI edit flow, not a new one-off mechanism.
- [ ] **Deterministic** - Mandatory deterministic review gate, per Pedro's explicit requirement: any AI-drafted (or AI-touched) paper version is stored with an explicit `requires_review: true`/draft status until `ledgerly validate <target>` (the existing deterministic engine, `engine.doc_validation.validate_document`) has been run against it and the user has explicitly reviewed its strengths/weaknesses/unsupported-sentence/citation-gap/confidence-score output — mirrors the existing citation-plan-insertion-review and cross-reference-candidate-review propose-then-apply pattern, applied here at whole-document scale. A paper must never silently become "final" just because AI produced it.
- [ ] **Deterministic** - Items above involving AI must not ship ahead of Phase 27 (AI Output Grounding and Anti-Hallucination Enforcement) — AI drafting an entire paper end-to-end is the highest-stakes place in this project for the Core Rule to fail, so this should be Phase 27's first real proof case, not built in parallel with looser grounding.
- [ ] **Deterministic** - Tests: the deterministic skeleton (items 6) is fully usable and produces a coherent document with zero AI configured; the review-gate flag cannot be cleared by anything other than an explicit user action following a `validate` run; an adversarial off-corpus paper-topic test (asking for a paper on a topic the workspace's sources don't cover) results in an explicit low-confidence/insufficient-evidence draft, never confident fabricated prose.
